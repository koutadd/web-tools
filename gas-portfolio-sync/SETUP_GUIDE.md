# request_id 移行ガイド

## 概要

行番号依存の更新処理を `request_id` ベースに移行する手順書。

---

## 前提確認

- 対象シート: `依頼シート`, `管理シート`
- 追加列: `request_id`（末尾に追加・既存列をズラさない）
- バックアップ: 作業前に必ずシートをコピーしておくこと

---

## Phase 1: スプレッドシートのバックアップ

1. Google Sheets を開く
2. 各対象タブを右クリック → 「シートをコピー」
3. コピー先: 同一スプレッドシートの末尾
4. コピー名: `依頼シート_bak_YYYYMMDD` 形式

---

## Phase 2: Apps Script のデプロイ

### 2-1. GAS プロジェクトを開く

1. スプレッドシートで「拡張機能」→「Apps Script」
2. 以下のファイルを追加/更新する:

| ファイル名 | 操作 |
|---|---|
| `Code.gs` | 内容を全置換（doGet 拡張版） |
| `doPost.gs` | 内容を全置換（request_id ベース版） |
| `RequestIdManager.gs` | 新規作成・貼り付け |

### 2-2. 権限の確認

- 「デプロイ」→「デプロイを管理」
- 実行者: 自分 (Me)
- アクセス: 全員 (Anyone)
- 新バージョンとして再デプロイ

---

## Phase 3: バックフィル実行

### 3-1. 既存行への request_id 一括付与

1. スプレッドシートのカスタムメニュー:
   **「🔑 request_id 管理」→「▶ バックフィル（全シート・完了含む）」**

2. 実行結果サマリを確認:
   ```
   ✅ 依頼シート: 付与XX件 / 既存維持0件 / 空行スキップXX件 / エラー0件
   ✅ 管理シート: 付与XX件 / 既存維持0件 / 空行スキップXX件 / エラー0件
   ```

3. エラーがある場合は GAS エディタの「ログを表示」で詳細確認

### 3-2. 未採番確認

**「🔍 未採番行の監査」** → 0行が未採番であることを確認

---

## Phase 4: トリガー設定

### 4-1. 定期補完トリガー

**「⏱ 定期補完トリガー設定」** を実行
→ 1時間ごとに `periodicBackfill` が動作

### 4-2. onEdit トリガー（手動設定が必要）

GAS エディタで:
1. 「トリガー」（時計アイコン）を開く
2. 「トリガーを追加」
3. 設定:
   - 実行する関数: `onEdit`
   - デプロイ: Head
   - イベントのソース: スプレッドシートから
   - イベントの種類: 編集時

> **注意**: `onEdit` は Simple Trigger（コードに書くもの）と Installable Trigger の2種類ある。
> スプレッドシートの外部からの書き込みには Installable Trigger が必要。

---

## Phase 5: アプリデプロイ

1. `aicare-portfolio.html` を更新版に差し替え
2. サーバー再起動 (`node server.js`)
3. ブラウザで動作確認（下記テスト項目を実施）

---

## テスト項目

### テスト 1: バックフィル確認
- [ ] 依頼シートの全データ行に `request_id` が付いている
- [ ] 完了ステータスの行にも付いている
- [ ] ヘッダー行には付いていない
- [ ] 完全空行には付いていない

### テスト 2: 新規行追加
- [ ] シートに手動で新行追加 → 保存後 `request_id` が自動付与される
- [ ] 既存行の `request_id` は変わらない

### テスト 3: 並び替え耐性
- [ ] シートの行を並び替える
- [ ] アプリからステータス更新 → 正しい行（request_id で特定）が更新される
- [ ] 行番号が変わっても更新先がズレない

### テスト 4: 他ユーザー追記
- [ ] 途中に行を手動挿入
- [ ] 元の依頼のステータス更新 → 正しい行が更新される

### テスト 5: 完了処理
- [ ] 依頼をドロワーで開いてステータスを「完了」に変更
- [ ] スプレッドシートの該当行（request_id 一致行）が更新される

### テスト 6: 格納処理
- [ ] 格納モーダルでファイルアップロード
- [ ] T列（制作完了データURL）に正しい行が書き込まれる
- [ ] 監査ログシートに記録が残る

### テスト 7: 不一致時の安全停止
- [ ] GAS 側でシートの店舗名を一時的に変更
- [ ] アプリからステータス更新 → エラーで停止（更新されない）
- [ ] 元に戻して再試行 → 成功

### テスト 8: ログ確認
- [ ] 「監査ログ」シートに success=true のログが記録される
- [ ] 失敗時は success=false + errorMessage が記録される

---

## ロールバック手順

### パターン A: アプリのみロールバック

1. `aicare-portfolio.html` を旧バージョンに戻す
2. サーバー再起動
3. GAS はそのままでも旧アプリは動作（rowId フォールバックが機能するため）

### パターン B: GAS のみロールバック

1. GAS エディタで「デプロイを管理」
2. 旧バージョン番号を選択してデプロイを切り替え
3. アプリ側の GAS_URL 更新不要（URLは変わらない）

### パターン C: スプレッドシートのロールバック

1. バックアップシート（`依頼シート_bak_YYYYMMDD`）を右クリック
2. 「シートをコピー」→「このスプレッドシートに」
3. コピーしたシートを「依頼シート」にリネーム
4. 元の「依頼シート」を「依頼シート_new」などにリネーム

> **注意**: request_id 列（末尾列）を削除しても既存データに影響なし。
> 既存列はズラしていないため、旧アプリの列番号参照も引き続き動作する。

### パターン D: 完全ロールバック

1. パターン C を実行（シートを旧状態に戻す）
2. パターン B を実行（GAS を旧バージョンに戻す）
3. パターン A を実行（アプリを旧バージョンに戻す）

---

## 未解決リスク

| リスク | 対策 |
|---|---|
| バックフィル前に誰かがアプリを操作した場合 | request_id 未付与行の更新は GAS 側でエラーを返す（fail closed）。バックフィル後に再試行を促す。 |
| 既存の格納フォルダ名（旧 rowId ベース）との不一致 | フォルダ名は引き続き `row_id` を使用（後方互換）。フォルダの中身は変わらない。 |
| GAS の 6 分実行時間制限 | バックフィルが大量行の場合は複数回実行。`added % 50 === 0` で 300ms 待機を入れてある。 |
| onEdit が取りこぼす追加方法 | periodicBackfill が 1 時間ごとに補完。重大な遅延はない。 |
| 監査ログシートが巨大化 | 月次で「監査ログ_YYYYMM」にアーカイブする運用を推奨。 |
